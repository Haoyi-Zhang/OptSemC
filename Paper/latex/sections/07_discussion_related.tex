\section{Discussion and Limitations}
\label{sec:discussion}

\textbf{Optimizer interfaces as semantic interfaces.}  Public optimizer claims should name action, guard, layer, placement, decision time, and evidence surface.  These fields do not expose private cost formulas; they keep adapters, remote sources, and service stages from collapsing into one phenomenon, a boundary already visible in federation~\cite{carey1995garlic} and polystores~\cite{stonebraker2015bigdawg}.

\textbf{What changes in an evaluation.}  The concrete change is the denominator of an optimizer study.  Before ranking latency, cardinality error, or plan quality, a study can test whether compared systems share the same public contract relation.  If a pushdown table equates connector delegation with runtime filtering, the study can retain the separating fields, narrow the claim, or report mixed optimizer phenomena.

\textbf{Using certificates with existing benchmarks.}  The certificate complements workload benchmarks by establishing a named denominator before performance measurement begins.  A study can still execute JOB, TPC-H, or a vendor suite, while OptSem-C states which optimizer contracts those runs may compare.  Safe retained fields give performance results grounded semantics; unsafe fields identify which layer, placement, or evidence surface must be separated before a database-system conclusion.

\textbf{Protocol, scope, and refresh.}  Every optimizer comparison is a projection: choose a vocabulary, extract public contracts, validate the denominator, compute the kernel, and publish retained fields or assumptions.  The evidence unit is a public contract, not inferred private state.  Without manuals, plans, diagnostics, connector behavior, or observable SQL behavior, a rule is outside scope; counts are lower bounds and later evidence becomes a source refresh.

\textbf{Reading a certificate.}  A repair certificate is a precondition on an evaluation denominator, not a performance metric.  Its witness, source spans, separating fields, and smaller-field counter-witness make the retained vocabulary reviewable before latency or plan-quality claims are compared.

\textbf{What the abstraction isolates.}  A repaired denominator is orthogonal to optimizer quality, runtime behavior, and cost-model comparison: it ensures that retained public contracts no longer collapse audited counter-witnesses.  This separation lets database researchers isolate semantic comparability from performance engineering before ranking systems or diagnosing private implementations.

\textbf{Threats and limits.}  Source selection can miss behavior vendors do not expose, so OptSem-C reports lower bounds over public contracts.  Human coding can misread a span; source identifiers, line ranges, digest prefixes, side-balanced support, and regenerated table-source checks mitigate that risk.  SQL execution validates visible plans, not private costs; closed systems need a separate admission policy.

\section{Related Work}
\label{sec:related}

\textbf{Optimizer architecture and extensible planning.}  OptSem-C's fields follow optimizer systems: System R exposes access paths and costing~\cite{selinger1979access}; Volcano~\cite{graefe1990volcano} and Cascades~\cite{graefe1995cascades} expose rules, layers, and search; and classical surveys frame costing and architecture~\cite{chaudhuri1998overview,ioannidis1996query}.  Calcite~\cite{calcite_docs}, DataFusion~\cite{datafusion_docs}, Trino pushdown~\cite{trino_pushdown_docs}, and BigQuery plans~\cite{bigquery_docs} show why placement and observability remain visible.  These systems also explain why one public label can hide several execution locations, planning layers, and evidence surfaces.

\textbf{Federated and cloud optimizers.}  Heterogeneous and cloud systems make those distinctions harder to treat as implementation detail.  Garlic~\cite{carey1995garlic}, BigDAWG~\cite{stonebraker2015bigdawg}, Spark SQL~\cite{armbrust2015spark}, Snowflake~\cite{dageville2016snowflake}, Redshift~\cite{gupta2015redshift}, and F1~\cite{shute2013f1} all expose boundaries between local planning, service execution, remote sources, and observable plan artifacts.  OptSem-C turns those boundaries into comparison fields rather than assuming that a shared feature name identifies the same optimizer contract.

\textbf{Benchmarks and evaluation denominators.}  Database benchmarks traditionally standardize workloads, metrics, or implementation rules, from systematic benchmarking practice~\cite{bitton1983wisconsin} to TPC-H~\cite{tpch}, TPC-DS~\cite{tpcds}, JOB~\cite{leis2015good}, and cardinality-estimation suites such as CardBench~\cite{chronis2024cardbench}.  OptSem-C is complementary: it standardizes neither the workload nor the ranking metric.  It tests whether the comparison vocabulary has already collapsed different public optimizer contracts before those workloads or metrics are applied.

\textbf{Equivalence, testing, and evidence.}  SQL-equivalence and containment work decide result preservation~\cite{chu2017cosette,chu2018usemiring,wang2024qed,chandra1977optimal,aho1979equivalences}; DBMS testers seek wrong answers or crashes~\cite{slutz1998massive,sqlsmith,rigger2020sqlancer,rigger2020norec}.  OptSem-C asks a different question: two systems may both return correct SQL answers while exposing different public contracts for where optimization occurs, when a decision is made, and what plan evidence is observable.  Provenance work separates support from behavior~\cite{green2007provenance,buneman2001why,green2011containment}; OptSem-C uses that separation to make optimizer-comparison denominators replayable.

\section{Conclusion}
OptSem-C makes optimizer comparisons replayable by turning coarse-label collisions into public-evidence certificates with hidden distinctions and repair fields.  Its central lesson is that optimizer studies need a stated denominator: which public contracts are equated, which fields are retained, and which counter-witnesses would invalidate the abstraction.
