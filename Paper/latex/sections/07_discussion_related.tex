\section{Discussion and Limitations}
\label{sec:discussion}

\textbf{Optimizer interfaces as semantic interfaces.}  A public optimizer claim should name the compared action, guard, layer, placement, decision time, and evidence surface.  These fields do not expose proprietary cost formulas; they prevent heterogeneous engines, adapters, remote sources, and service-side stages from being treated as one phenomenon.  This matters in Garlic-style federation~\cite{carey1995garlic} and BigDAWG-style polystores~\cite{stonebraker2015bigdawg}, where one user operation crosses engine and source boundaries.

\textbf{Protocol, scope, and validity.}  Every optimizer comparison is a projection: choose a vocabulary, extract public contracts, validate the denominator, compute the kernel, and publish retained fields or assumptions.  The evidence unit is a public contract, not inferred private state; absent manuals, plans, diagnostics, connector behavior, or observable SQL behavior, a rule is outside scope unless it changes that contract.  Because documentation can be incomplete or stale, OptSem-C separates source digests, version scope, and visible-plan checks; a document--plan disagreement is a contract-evidence conflict, not a private-truth correction.  Motif crosswalks, source leave-one-out tests, feature holdouts, engine-family stress, and real-engine SQL checks mitigate circularity by changing the denominator, support, or execution surface.

\textbf{Using repair certificates.}  A repair certificate changes how comparisons are stated.  Instead of saying that two systems both ``support pushdown'' or ``use adaptive optimization,'' a study can state the certified frontier: local versus delegated execution, logical versus physical planning, compile-time versus runtime choice, and plan versus contract evidence.  Version scope and evidence digests make source refresh an incremental audit.

\section{Related Work}
\label{sec:related}

\textbf{Optimizer architecture and extensible planning.}  Classical optimizers separate relational equivalence, access paths, costing, and search: System R exposes access-path selection~\cite{selinger1979access}, Chaudhuri surveys optimizer pipelines~\cite{chaudhuri1998overview}, and Ioannidis surveys cost assumptions~\cite{ioannidis1996query}.  Volcano structures rule-driven processing~\cite{graefe1990volcano}; the optimizer-generator paper generalizes construction~\cite{graefe1993volcano}; and Cascades casts search as tasks~\cite{graefe1995cascades}.  Calcite traits and adapters~\cite{begoli2018calcite,calcite_docs}, Orca modules~\cite{soliman2014orca}, DataFusion plans~\cite{datafusion_docs}, and Presto settings~\cite{presto_docs} make public-comparison fields observable.

\textbf{Equivalence, testing, and provenance.}  Cosette~\cite{chu2017cosette}, semiring-based SQL equivalence~\cite{chu2018usemiring}, and QED~\cite{wang2024qed} decide result preservation, while NoREC exposes optimizer bugs~\cite{rigger2020norec}.  Chandra and Merlin~\cite{chandra1977optimal} and Aho et al.~\cite{aho1979equivalences} characterize containment and expression equivalence.  OptSem-C addresses a different question: result-equivalent plans can still expose different public optimizer evidence.  Its evidence model follows provenance semirings~\cite{green2007provenance}, why/where provenance~\cite{buneman2001why}, provenance surveys~\cite{cheney2009provenance}, and annotated-query containment~\cite{green2011containment}.

\textbf{Optimizer benchmarks and adaptive systems.}  JOB makes the plan-quality denominator explicit~\cite{leis2015good}, later JOB analysis exposes optimizer behavior~\cite{leis2018looking}, and exact-cardinality testing isolates choices~\cite{chaudhuri2009exact}.  Progressive optimization~\cite{markl2004progressive}, mid-query reoptimization~\cite{kabra1998reoptimization}, adaptive query processing~\cite{deshpande2007adaptive}, and eddies~\cite{avnur2000eddies} move decisions to runtime.  Learned optimizers choose or rank plans~\cite{marcus2019neo,marcus2021bao,zhu2023lero}, while Learned Cardinalities~\cite{kipf2019learned}, NeuroCard~\cite{yang2020neurocard}, and DeepDB~\cite{hilprecht2020deepdb} model distributions.  OptSem-C asks whether compared behavior has the same public contract denominator.

\textbf{Workload and system benchmarks.}  TPC-H~\cite{tpch} and TPC-DS~\cite{tpcds} define decision-support denominators; SSB~\cite{oneil2009ssb}, LDBC~\cite{erling2015ldbc}, LakeBench~\cite{deng2024lakebench}, and CardBench~\cite{chronis2024cardbench} target warehouses, graph analytics, data-lake joins, and learned cardinalities.  Wisconsin~\cite{bitton1983wisconsin}, AS3AP~\cite{turbyfill1989as3ap}, and ClickBench~\cite{clickbench} add portability and scale views.  OptSemBench-C keeps that discipline but checks optimizer-facing motifs against public contracts.

\section{Conclusion}
A coarse optimizer comparison can declare equivalence where public contracts disagree.  OptSem-C formalizes that failure with finite projections over 287 rules and 4,216 probes.  In this corpus, layer+placement separates all 498 headline witnesses, and every unsafe retained-field region has a concrete counter-witness.
